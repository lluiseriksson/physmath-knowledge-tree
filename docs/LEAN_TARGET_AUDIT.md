# Lean Target Audit Center

The Lean Target Audit Center is a local-first companion to the canonical research graph. It turns each node's candidate Lean imports, declarations and bounded formalization targets into an explicit review queue without changing graph confidence, theorem status or canonical JSON.

Open `formalization.html` from the deployed site or local development server.

## Purpose and boundary

The graph intentionally stores candidate `lean.imports`, `lean.declarations` and `lean.targets`. Those fields are useful only when their relationship to a concrete Lean/mathlib toolchain is visible. The audit center records the missing operational layer:

- which candidate import or declaration was checked;
- which Lean/mathlib revision was used;
- whether the candidate resolved, was missing, was renamed or was blocked;
- which replacement name should be tested next;
- what command or finite experiment was performed.

A local `verified` record means only that a human recorded a successful name/import check against the stated toolchain. It does not prove the node summary, promote an evidence label or certify a research programme.

## Canonical input

The browser application and CLI read:

```text
graph/nodes/core.json
```

The normalizer requires stable unique node IDs, bounded text and array-shaped Lean metadata. It produces deterministic audit items of three types:

```text
Lean audit catalog
├── import       candidate module name
├── declaration  candidate constant/theorem/type name
└── target       bounded formalization objective
```

Every item ID is a collision-free JSON tuple of canonical node ID, item type and exact value. When canonical metadata changes, stale local records are discarded instead of being attached to a different candidate.

## Local ledger

Each candidate can have one browser-local record:

```text
record
├── item_id
├── status: unreviewed | verified | missing | renamed | blocked
├── checked_at
├── toolchain
├── replacement
├── notes
└── updated_at
```

`renamed` can point to a replacement import or declaration. `blocked` records an environmental or prerequisite obstruction. `missing` records a failed lookup under the stated toolchain. None of these statuses mutate canonical graph data.

## Worklist priority

The default queue is deterministic. It gives unresolved declarations more weight than imports, and imports more weight than prose targets. Reused candidates and stronger graph evidence receive additional triage weight. The score is only a work-order heuristic; it is not mathematical confidence.

The queue can be filtered by:

- free text over node, candidate, replacement and notes;
- local audit status;
- item type;
- graph node kind;
- graph evidence label.

## Reproducible Lean probes

Select imports, declarations or targets and choose **Preview Lean probe**. The generator:

1. gathers the selected nodes' canonical imports;
2. applies probe-safe recorded replacements for `renamed` candidates;
3. omits `blocked` or syntactically unsafe names with explicit comments;
4. emits `#check` commands for declaration candidates;
5. preserves prose targets as comments;
6. states that compilation checks names only.

Example:

```lean
/-
Generated by PhysMath Knowledge Tree Lean Target Audit.
This file checks that selected imports and declaration names resolve.
Compilation does not validate the surrounding mathematical claim.
-/

import Mathlib.CategoryTheory.Category.Basic

set_option autoImplicit false

-- Node: Category theory (domain.category_theory)
#check CategoryTheory.Category
#check CategoryTheory.Functor
```

Run a downloaded probe from the repository root with:

```bash
lake env lean PhysMathLeanProbe.lean
```

A successful command is evidence about the named imports and constants in that exact environment. Record the full toolchain or mathlib revision before marking an item verified.

## CLI probe generation

The same deterministic generator is available without the browser:

```bash
npm run lean:probe -- -- --output PhysMathLeanProbe.lean
npm run lean:probe -- -- --node domain.category_theory --type declaration
npm run lean:probe -- -- --node problem.riemann_hypothesis --toolchain leanprover/lean4:v4.31.0
npm run lean:probe -- -- --ledger physmath-lean-audit-ledger.json --output ReviewedProbe.lean
```

Supported options are `--node`, `--type`, `--toolchain`, `--ledger` and `--output`. Multiple `--node` and `--type` arguments are allowed. `--ledger` applies exported `renamed` and `blocked` records while generating the probe. The CLI writes only the generated probe; it does not mark any ledger record verified or edit canonical data.

## Import, export and review packets

`Export ledger JSON` writes the complete local ledger. Import validates:

- application and schema identity;
- an 8 MB size limit;
- JSON-compatible filename and media type when supplied by the browser;
- bounded record counts and text fields;
- exact item IDs from the current canonical catalog.

Merge mode keeps the newest `updated_at` record per item. Replace mode replaces the local ledger after validation.

Selected items can also be exported as:

- a JSON packet containing catalog metadata, records and the generated probe;
- a Markdown packet containing a review table and fenced Lean source.

These packets are deliberate handoff artifacts for repository review. They are not canonical verification records by themselves.

## Privacy and failure behavior

No account, analytics, cookie or remote synchronization is introduced. Audit records remain in browser storage unless explicitly exported. Storage denial or quota exhaustion does not prevent read-only catalog exploration or probe generation.

## Verification

The pure audit model is covered by `tests/lean-target-audit.test.mjs`, including malformed metadata, stale records, merge semantics, probe safety, replacement handling and portable packets. `tests/lean-target-audit-product.test.mjs` checks the public page, CSP, CLI and read-only boundary. A focused dependency-free Chromium smoke exercises queue loading, filtering, record persistence and probe generation.
