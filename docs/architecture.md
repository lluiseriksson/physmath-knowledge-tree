# Architecture

PhysMath Knowledge Tree combines two static applications, one canonical research graph and a small Lean package. The design is dependency-light, deterministic, offline-capable and deployable without a backend.

## Product surfaces

### Research graph (`index.html`)

The home page loads canonical JSON from `graph/`. `src/research-app.js` coordinates filters, curated collections, SVG/list rendering, node dossiers, path search, export, bridge-card scaffolding, URL state, language and theme. It never duplicates the canonical graph in JavaScript.

`src/lib/research-graph.js` contains pure indexing, search, neighborhood, shortest-path, induced-subgraph and deterministic force-layout functions. `src/lib/research-i18n.js` contains interface translations. `src/research.css` owns visual tokens and responsive behavior.

### Learning map (`learning.html`)

The learning experience uses the catalog in `src/data/topics.js`. Pure graph, search, persistence and URL modules derive prerequisites, paths, readiness and local learner state. `src/app.js` orchestrates rendering and browser interactions.

The two applications share manifest, service worker, icons, security policy and build pipeline, while keeping their domain logic separate.

## Canonical research data

`graph/index.json` declares canonical paths, schema paths, roots, entry points and generated statistics. Nodes, edges, research moves and collections are validated independently and together.

JSON is canonical. Files in `views/` are deterministic projections generated by `scripts/generate-views.mjs`; CI fails when they are stale. `graph/audit.json`, `graph/reference-registry.json` and `docs/GRAPH_AUDIT.md` are generated evidence/integrity reports, not alternate graph sources. Evaluation scenarios are canonical fixtures in `evaluation/scenarios.json`; their reports are deterministic projections.

## Lean layer

The Lean package models confidence, node kinds, edge kinds, targets and typed example cards. It is an ontology spine and formal metadata layer, not a proof of the scientific claims in the graph. Lean and web validation run in separate CI jobs.

## Security boundaries

Shipped source data is trusted only after CI validation. Browser storage, URL parameters and imported learning-progress files are untrusted and sanitized. Canonical graph content is rendered through element creation and `textContent`; it is not interpreted as HTML.

Runtime resources are same-origin. Progress imports are size/type bounded and storage failures are non-fatal. There are no analytics, accounts, remote fonts or third-party scripts. A restrictive CSP is present in HTML and a generated `_headers` file supports hosts that consume it.

## Offline layer

`sw.js` precaches both application shells, a dedicated offline page and canonical graph files. Navigation uses a route-aware network-first strategy; cached assets are served immediately and refreshed in the background. The cache version is checked against `package.json`. User progress remains in local storage, not cache storage.

## Build and quality gate

`scripts/build.mjs` creates `dist/` from an explicit allowlist and writes a deterministic SHA-256 manifest. The quality gate checks:

1. JavaScript syntax;
2. research graph integrity and schema parseability;
3. curation provenance, text/image anchors and generated deletion report;
4. learning graph integrity and acyclicity;
5. local links, CSP, static accessibility and PWA invariants;
6. generated-view, evidence-audit, reference-registry and evaluation freshness;
7. SHA-pinned workflow actions, least-privilege checkout and explicit permissions;
8. formatting invariants and unit tests;
9. static build plus per-file SHA-256 verification.

GitHub Pages receives only `dist/`. CodeQL scans JavaScript separately. The Lean job uses the pinned mathlib cache, warnings-as-errors and `mk_all` root-import coverage. The bundled `leanchecker` step is kept out of the required CI path because it currently cancels on GitHub-hosted runners after a successful Lake build.

## Why no frontend framework

The deployed interface is small enough for browser primitives, and the core algorithms are easier to test as pure modules. Avoiding runtime dependencies reduces supply-chain exposure and upgrade churn. A future dependency should be added only after demonstrating a measurable benefit that outweighs security, accessibility and maintenance costs.

## Curation boundary

`curation/` stores hashes and review decisions, never raw private source files. Text claims are anchored by line ranges; PNG claims use bounded pixel regions with optional caption, alt text and transformation history. Extraction status and user approval are separate fields, so automation cannot infer deletion permission from curation alone. `scripts/verify-curation-source.mjs` can recheck a retained local original without copying it into the repository.
